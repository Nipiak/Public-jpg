📌 Qu'est-ce que Microsoft Intune ?

Microsoft Intune est une solution de gestion unifiée des endpoints (UEM — Unified Endpoint Management) entièrement hébergée dans le cloud Azure. Elle permet à une organisation de gérer, configurer, sécuriser et monitorer tous ses appareils (PC Windows 10/11, Mac, iPhone, iPad, Android) et leurs applications depuis un seul portail web — sans aucune infrastructure serveur à maintenir sur site.

  • Gestion complète des devices : Windows, macOS, iOS/iPadOS, Android
  • Apps Win32, LOB, Microsoft 365 Apps, Store — tout depuis un seul portail
  • Sécurité : BitLocker, Defender, LAPS, ASR, Firewall — configurable à distance
  • Couplé à Entra ID pour le Conditional Access et la conformité

Qu'est-ce qu'un "Device" dans Intune ?

Un device (appareil) dans Intune est tout équipement physique ou virtuel qui s'est inscrit (enrollé) dans le service Intune et qui reçoit les policies, apps et configurations de l'organisation. Intune les gère via le protocole MDM (Mobile Device Management) qui communique en HTTPS sur le port 443.

MDM vs MAM — La distinction fondamentale

MDM — Mobile Device Management

Gère le DEVICE entier

  • ✓Enrollment obligatoire
  • ✓Contrôle total du device
  • ✓Wipe/Lock à distance
  • ✓Toutes les policies
  • !Admin peut voir infos device
MAM — Mobile Application Management

Gère les APPLICATIONS seulement

  • ✓Pas d'enrollment requis
  • ✓Respect vie privée total
  • ✓Idéal BYOD smartphones
  • ✓Selective Wipe possible
  • ✗Pas de contrôle du device
MDM + MAM combinés

Contrôle total + protection apps

  • ✓Device géré entièrement
  • ✓Apps corpo protégées
  • ✓Devices corporate
  • ✓Sécurité maximale
  • !Plus intrusif pour l'user

Qu'est-ce que Windows Autopilot ?

Service cloud Microsoft qui configure automatiquement un PC Windows lors de sa première utilisation (OOBE — Out-of-Box Experience) sans aucune intervention IT physique. Le PC est livré directement à l'utilisateur qui n'a qu'à l'allumer, se connecter avec son compte professionnel — tout se configure automatiquement.

  • Zéro intervention IT physique sur le PC (livraison directe à domicile possible)
  • Jointure Entra ID + enrollment Intune automatiques pendant l'OOBE
  • Apps, policies, nom du PC appliqués sans technicien
  • Remplace MDT, SCCM OSD, Ghost pour les nouveaux déploiements

Utiliser seulement si : L'entreprise a des legacy apps nécessitant absolument l'AD on-prem (Kerberos, NTLM, partages SMB non migrables). Si pas de contrainte → préférer Entra ID Join pur (plus simple).

Enrollment Status Page (ESP)

Le gardien du bureau Windows — configuration, phases, apps bloquantes et diagnostic

Qu'est-ce que l'ESP ?

L'Enrollment Status Page est la page de progression qui s'affiche pendant le provisionnement Autopilot. Elle bloque l'accès au bureau tant que les apps critiques et les policies ne sont pas installées. Sans ESP correctement configurée, l'utilisateur arrive sur un bureau vide — ou pire, accède au bureau avant que les apps de sécurité soient là.

Diagnostic — Pourquoi l'ESP ne bloque pas le bureau ?

❌ Cause 1 (la plus fréquente) Apps absentes de la liste bloquante ESP
Le problème

Tes apps sont en Required dans Intune → elles s'installeront. Mais l'ESP ne sait pas qu'elle doit attendre ces apps spécifiques → elle laisse passer au bureau.

La solution
ESP → Settings → Edit "Bloquer jusqu'à installation des apps requises" → Oui → Sélectionner les applications → [Ajouter tes 2 apps ici] ✅
❌ Cause 2 Apps en User context → Phase 3 seulement
Le problème

Install behavior = User → l'app est en Phase 3 (Account setup, APRÈS le login utilisateur). La Phase 2 (Device setup, sans login) ne la voit pas → bureau accessible avant installation.

La solution
Apps → [Ton app Win32] → Program → Edit Install behavior : System ← Phase 2 ✅ (avant login) # PAS "User" ← Phase 3 seulement
❌ Cause 3 ESP assignée au mauvais groupe ou pas du tout
Le problème

L'ESP n'est pas assignée au groupe contenant ton device. Ou pire : assignée à un groupe d'utilisateurs alors qu'elle doit cibler des devices.

La solution
ESP → Assignments → Add group → DEVICE group # Même groupe que le profil Autopilot # Jamais un groupe d'utilisateurs !
❌ Cause 4 Option "Bloquer" désactivée dans l'ESP
Le problème

L'ESP s'affiche et montre la progression, mais l'option de blocage est sur Non → l'utilisateur peut cliquer sur "Ignorer" et accéder au bureau.

La solution
ESP → Settings → Edit "Bloquer l'utilisation de l'appareil jusqu'à ce que toutes les apps et profils soient installés" → Oui Autoriser bypass si erreur → Non

Le MAM sans enrollment protège les données DANS les applications (Outlook, Teams, OneDrive) sans gérer le device entier. L'IT ne voit pas les données personnelles, les apps perso, les photos. C'est la solution idéale pour le BYOD car elle respecte totalement la vie privée de l'employé.

  • Pas d'enrollment MDM requis — le device reste sous contrôle de l'utilisateur
  • Selective Wipe possible : efface uniquement les données corpo, laisse le perso intact
  • L'activation se fait au moment où l'user se connecte avec son compte Entra ID dans l'app

Pourquoi LAPS est essentiel ?

Conformité — Configuration complète

Policy de conformité, actions, grace period, lien avec Conditional Access

Règle fondamentale : Conformité = JUGE (pas configurateur)

  • La conformité ÉVALUE l'état du device → conforme ou non conforme
  • Elle ne CONFIGURE rien (BitLocker doit être activé par un profil de config)
  • Seule → ne bloque rien. Avec CA → bloque l'accès M365 si non conforme
  • Grace period → délai avant marquage officiel non conforme

EXE vs MSI vs INTUNEWIN — La différence fondamentale

  • .exe : Installateur non standardisé. NE peut PAS être uploadé directement dans Intune → doit être converti en .intunewin. Paramètres silencieux variables selon l'éditeur.
  • .msi : Installateur standardisé Microsoft. Peut être uploadé directement (LOB App). Paramètres universels : /quiet /norestart. GUID unique pour la détection.
  • .intunewin : Format natif Intune. Chiffré AES-256, compressé. Créé avec IntuneWinAppUtil.exe. Compatible .exe et .msi. C'est le format à utiliser pour les Win32 Apps.

🔑 Primary Refresh Token (PRT)

Token spécial délivré par Entra ID lors de la connexion Windows. Stocké dans le TPM. Permet le SSO automatique vers tous les services M365 sans re-saisir le mot de passe.

📋 Entra ID Registered vs Joined

Registered = BYOD, compte perso ajouté → MAM seulement. Joined = Device corporate rejoint Entra ID → MDM complet + SSO natif + WHfB.

⏱️ Device Token Lifetime

Le PRT est renouvelé toutes les 4 heures si le device est actif. Sans renouvellement (device éteint >14j) → re-connexion complète requise. Configurable via CA Sign-in frequency.

Certificats SCEP & PKCS dans Intune

PKI, NDES, connecteur certificat — authentication sans mot de passe

Pourquoi des certificats dans Intune ?

  • Authentification Wi-Fi 802.1x sans mot de passe — le certificat prouve l'identité du device
  • Authentification VPN avec certificat client — plus sécurisé qu'un mot de passe partagé
  • Chaque device reçoit son propre certificat unique — si compromis, on révoque seulement le sien
  • L'utilisateur ne voit rien — tout est automatique et transparent
📜

SCEP vs PKCS — Quelle différence ?

🔵 SCEP — Simple Certificate Enrollment Protocol

Chaque device génère sa propre paire de clés privée/publique. La clé privée ne quitte jamais le device.

  • ✓Clé privée unique par device
  • ✓Plus sécurisé — clé non exportable
  • ✓Idéal Wi-Fi 802.1x et VPN
  • !Nécessite NDES on-prem ou Cloud PKI
  • !Infrastructure PKI + connecteur requis

🟠 PKCS — Public Key Cryptography Standards

Intune génère le certificat sur le serveur CA et le déploie sur le device. La clé est créée côté serveur.

  • ✓Pas besoin de NDES
  • ✓Plus simple à configurer
  • ✓Idéal email S/MIME, apps
  • ✗Clé privée transite par Intune
  • ✗Moins sécurisé que SCEP pour l'auth

  • Assume breach : Minimiser le rayon d'explosion si compromission — segmentation, chiffrement, MDE

Comment aborder les scénarios complexes à l'exam

  • Lire la question entière avant de répondre — les pièges sont souvent dans les détails (licence, OS, type de device)
  • Identifier les mots-clés : "sans enrollment", "corporate", "BYOD", "automatiquement", "sans admin"
  • Éliminer les réponses impossibles (mauvaise licence, mauvais OS, wrong approach)
  • Vérifier que la réponse résout TOUTES les contraintes, pas seulement les principales
🎓

Scénario 1 — Déploiement hybride multi-contraintes

🎯 Contexte : Votre entreprise a 500 PC Windows 11 gérés par SCCM. Vous devez activer le Conditional Access basé sur la conformité des devices, sans réenroller les PC et sans les migrer vers Entra ID Join pur. Les utilisateurs doivent accéder à Exchange Online seulement si leur device est conforme.

Solution étape par étape

1. Co-gestion SCCM + Intune

Les PC restent joints à l'AD on-prem (Hybrid Join). Activer Cloud Attach dans SCCM + Co-gestion. Les PC s'enrollent dans Intune sans réinstallation.

2. Basculer le workload "Compliance" vers Intune

Dans SCCM → Cloud Attach → Co-gestion → Compliance policies → glisser le curseur vers Intune.

3. Créer la Compliance Policy dans Intune

Devices → Compliance policies → Windows → critères : BitLocker, Defender, OS version min.

4. Créer la CA Policy dans Entra ID

Users : All → Exchange Online → Grant : Require compliant device. Exclure break-glass.

5. Résultat

PC non conforme (BitLocker désactivé) → CA bloque Exchange Online → user doit corriger → PC conforme → accès rétabli.

Pièges à éviter : Oublier d'activer le workload Compliance dans SCCM (résultat : les policies Intune sont ignorées). Oublier le compte Break-glass dans l'exclusion CA. Ne pas créer la CA policy (la conformité seule ne bloque rien).
🎓

Scénario 2 — Sécurisation BYOD international

🎯 Contexte : Des commerciaux utilisent leur iPhone personnel pour accéder à Outlook. Vous devez : (1) protéger les données corpo sans gérer le device, (2) bloquer l'accès depuis des pays non autorisés, (3) forcer un PIN sur Outlook, (4) permettre un Selective Wipe au départ.

Protéger les données sans gérer le device
→
App Protection Policy (MAM) — pas d'enrollment requis
Bloquer l'accès depuis certains pays
→
Named Location "Pays bloqués" + CA Policy → Block
Forcer PIN sur Outlook iOS
→
App Protection Policy → Access requirements → PIN requis → 6 chiffres
Selective Wipe au départ
→
Retire (MAM) → efface données corpo, conserve données perso
Forcer l'utilisation de l'APP (pas Outlook perso)
→
CA → Grant → Require app protection policy
Point critique : L'App Protection Policy doit être assignée à un groupe d'UTILISATEURS (pas de devices). Si assignée à un groupe de devices → elle ne s'applique jamais.
🎓

Scénario 3 — Déploiement Autopilot avec contraintes réseau

🎯 Contexte : Vous déployez 200 PC via Autopilot User-Driven. L'ESP contient 15 apps bloquantes. Les utilisateurs se plaignent que le déploiement prend 45 minutes et certains PCs se bloquent avec un écran rouge. Que faire ?

1. Analyser les logs ESP

IntuneManagementExtension.log → chercher quelle app bloquante échoue → ExitCode non-0

2. Vérifier les apps bloquantes

Toutes les apps bloquantes dans l'ESP sont-elles aussi assignées en Required en Device context ? Une seule en User context peut faire bloquer Phase 2 → écran rouge.

3. Augmenter le timeout ESP

ESP → Edit → Timeout : augmenter à 120 minutes. 45 min OK, mais si une app tarde sur réseau lent → timeout → rouge.

4. Solution optimale → White Glove

Activer White Glove dans le profil Autopilot. IT installe les 15 apps en Phase 1 en entrepôt. L'utilisateur n'attend plus que 5-10 min (Phase 2 uniquement).

5. Alternative → Réduire les apps bloquantes

Ne mettre que les apps CRITIQUES en bloquant (3-5 max). Les autres en Required sans bloquer → s'installent en arrière-plan après accès au bureau.

Windows Delivery Optimization

Économiser la bande passante lors des mises à jour Windows en partageant les téléchargements

Qu'est-ce que la Delivery Optimization ?

  • Technologie Windows permettant aux devices de se partager les mises à jour entre eux (peer-to-peer) plutôt que de télécharger depuis Microsoft chacun
  • Un device qui a déjà téléchargé un patch peut le partager aux autres devices du réseau
  • Économise jusqu'à 90% de la bande passante sortante sur les réseaux d'entreprise
  • Configurable via Intune → Settings Catalog → Delivery Optimization
📊

Schéma — Sans vs Avec Delivery Optimization

❌ Sans Delivery Optimization
💻 PC-001 → télécharge 500 Mo depuis Azure
💻 PC-002 → télécharge 500 Mo depuis Azure
💻 PC-003 → télécharge 500 Mo depuis Azure
100 PC × 500 Mo = 50 Go de bande passante !
Internet saturé pendant les patches
✅ Avec Delivery Optimization
💻 PC-001 → télécharge 500 Mo depuis Azure
💻 PC-002 → reçoit de PC-001 (LAN) ♻️
💻 PC-003 → reçoit de PC-001 ou 002 ♻️
100 PC → peut-être 1-2 Go seulement depuis Azure
98% de bande passante économisée !

Bulk Remote Actions — Actions en masse

Pourquoi les bulk actions ?

Certaines situations nécessitent d'agir sur des centaines de devices en même temps : campagne de sécurité, réponse à un incident, décommissionnement massif. Intune permet des actions en masse sur une sélection de devices.

2

Via PowerShell — Actions en masse sur un groupe

# Exemple : Sync tous les devices d'un groupe Connect-MgGraph -Scopes "DeviceManagementManagedDevices.ReadWrite.All" $groupId = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx" $devices = Get-MgGroupMember -GroupId $groupId -All foreach ($d in $devices) { try { Sync-MgDeviceManagementManagedDevice -ManagedDeviceId $d.Id Write-Host "✅ Sync: $($d.AdditionalProperties.displayName)" } catch { Write-Warning "❌ Erreur: $($d.AdditionalProperties.displayName)" } Start-Sleep -Milliseconds 500 # Éviter le rate limiting }

Rotation des clés BitLocker — Remote Action

Pourquoi faire pivoter les clés BitLocker ?

Après qu'une clé de récupération a été utilisée (par le support pour débloquer un PC), il est recommandé de la faire pivoter — une nouvelle clé est générée et sauvegardée dans Entra ID. L'ancienne clé (potentiellement notée par le technicien de support) n'est plus valide.

1

Rotation manuelle depuis Intune

intune.microsoft.com → Devices → All devices → [Sélectionner le device] → Rotate BitLocker keys # Une nouvelle clé de 48 chiffres est générée # L'ancienne clé est invalidée # La nouvelle clé est visible dans Entra ID → [Device] → Recovery keys
2

Rotation automatique après utilisation (recommandé)

# Dans la policy BitLocker Intune Endpoint security → Disk encryption → [Policy] BitLocker base settings : Recovery password rotation : Rotation enabled for Azure AD-joined devices # Dès qu'une clé est utilisée → nouvelle clé générée automatiquement
3

Rotation en masse via PowerShell

# Rotation des clés BitLocker sur tous les devices d'un groupe $devices = Get-MgGroupMember -GroupId "[GUID]" -All foreach ($d in $devices) { Invoke-MgRotateManagedDeviceBitlockerKey -ManagedDeviceId $d.Id Write-Host "🔐 Clé pivotée: $($d.AdditionalProperties.displayName)" }

Microsoft Copilot for Security dans Intune

Depuis janvier 2026, le programme MD-102 mentionne explicitement Copilot for Security. L'IA générative est intégrée dans Intune (Device summary AI, Policy insights), Defender XDR et Entra ID. Cas d'usage : analyser des incidents, générer des scripts KQL, obtenir des résumés de device.

Top 15 pièges classiques de l'exam MD-102

1
Compliance Policy ≠ Configuration ProfilePiège fréquent
La Compliance JUGE (évalue). Le Config Profile CONFIGURE (applique des paramètres). Ne jamais confondre les deux. Si la question demande "comment activer BitLocker" → Config Profile. "Comment vérifier si BitLocker est activé" → Compliance Policy.
❌ Erreur type : "Pour forcer le chiffrement BitLocker, créer une Compliance Policy avec BitLocker=Required." → FAUX. La Compliance vérifie, elle ne force pas. Il faut un Config Profile (Endpoint Security → Disk Encryption) pour activer BitLocker.
2
Sans Compliance Policy = Non-Compliant (depuis 2022)Changement 2022
Depuis une mise à jour Intune 2022, un device sans aucune compliance policy assignée est marqué Non-Compliant par défaut. Ce comportement contre-intuitif est souvent testé à l'exam.
✅ Règle : Toujours assigner une compliance policy, même basique, à tous les devices. Sinon ils seront bloqués par CA si "Require Compliant Device" est configuré.
3
Ordre SCEP : Root CA → SCEP → Wi-Fi (impératif)Ordre critique
L'ordre de déploiement des profils certificats est non négociable. Si vous déployez le profil Wi-Fi avant que le device ait reçu le certificat SCEP, l'authentification 802.1x échoue.
❌ Inverser l'ordre = Wi-Fi KO garanti sur tous les devices. Mémoriser : "RCA avant SCEP avant Wi-Fi".
4
Compliance ne bloque pas — c'est le CA qui bloqueArchitecture
Un device Non-Compliant n'est pas automatiquement bloqué. C'est le Conditional Access qui bloque, en lisant le statut de conformité. Sans CA configuré avec "Require Compliant", un device non-conforme accède quand même à M365.
✅ Les deux travaillent ensemble : Compliance Policy (juge) + CA Policy "Require Compliant" (bloque).
5
MDM vs MAM — lequel choisir pour BYOD ?BYOD
Pour un device personnel (BYOD) où l'utilisateur ne veut pas que l'IT touche à son device : MAM (App Protection Policy). MDM = enrollment complet, droit au wipe total = inacceptable pour BYOD.
✅ Règle : BYOD = MAM-WE (sans enrollment). Device corporate = MDM. Les deux peuvent coexister.
6
Break-glass OBLIGATOIRE avant d'activer le CASécurité critique
Activer une CA policy sans compte break-glass = risque de lockout total du tenant. Le compte break-glass doit être exclu de toutes les CA policies et ses credentials stockés hors ligne.
❌ Erreur type : Activer CA "Block all" sans exclusion break-glass = plus personne n'accède au tenant.
7
AADJ vs HAADJ — quand utiliser quoi ?Enrollment
AADJ = cloud-only, Autopilot User-Driven, pas d'AD on-prem requis. HAADJ = besoin de GPO on-prem, AD existant, Entra Connect requis, Autopilot hybride.
✅ Règle : Si l'entreprise a des applications qui nécessitent un domaine on-prem (Kerberos) → HAADJ. Sinon → AADJ préféré.
8
IME check-in = 8 heures (pas immédiat)Délais
En fonctionnement normal, l'IME (et les compliance policies) se synchronisent toutes les 8 heures. Pour forcer immédiatement : Company Portal → Sync, ou Intune → Device → Sync.
❌ Erreur type : "La politique s'applique immédiatement" → FAUX. Délai normal 8h. Sauf dans la première heure post-enrollment (5 min).
9
Autopilot Self-Deploying = TPM 2.0 obligatoireAutopilot
Le mode Self-Deploying Autopilot nécessite obligatoirement TPM 2.0 car il s'authentifie via TPM (pas d'utilisateur présent). User-Driven peut fonctionner sans TPM (mais WHfB ne sera pas disponible).
✅ Mémo : Self-Deploying = sans user = TPM obligatoire. User-Driven = user présent = TPM recommandé mais non bloquant.
10
ASR en mode Block sans Audit préalable = risquesSécurité
Activer les règles ASR directement en mode Block sans passer par Audit peut bloquer des applications légitimes (faux positifs). La bonne pratique = Audit → analyser les logs → Block progressif.
✅ Ordre correct : Audit (mesurer impact) → Warn (alerter l'user) → Block (bloquer). Jamais Block direct en production.
11
Selective Wipe ≠ Full Wipe sur device BYODBYOD
Selective Wipe (via MAM) = efface uniquement les données d'entreprise dans les apps gérées. Full Wipe = reset total du device (factory reset). Sur un BYOD, seul le Selective Wipe est éthiquement acceptable.
❌ Full Wipe sur BYOD = perte de toutes les données personnelles de l'utilisateur. Utiliser Selective Wipe / Retire.
12
Quality Update deferral max = 35 jours (pas illimité)Updates
On ne peut pas différer les Quality Updates de plus de 35 jours. Feature Updates = 365 jours maximum. Au-delà du maximum, Windows Update force la mise à jour.
✅ Mémo : Quality (sécurité mensuelle) = max 35j. Feature (mise à niveau OS) = max 365j.
13
Scope Tags ≠ Filters — deux choses différentesAdministration
Scope Tags = contrôlent la visibilité RBAC (quel admin voit quoi). Filters = contrôlent l'affectation des policies (quels devices reçoivent quoi).
✅ Mémo : Scope Tags = RBAC visibility. Filters = Assignment targeting. Complémentaires, pas synonymes.
14
dsregcmd /status = commande de diagnostic #1Diagnostic
La commande dsregcmd /status est systématiquement citée dans les questions de troubleshooting Intune/Entra ID. Vérifier : AzureAdJoined + MdmEnrolled + MdmUrl présent.
✅ Résultat attendu sur un device enrollé : AzureAdJoined : YES — MdmEnrolled : YES — WorkplaceJoined : NO
15
LAPS prévient le lateral movementSécurité
Le lateral movement = un attaquant compromet 1 PC, récupère le mdp admin local, et l'utilise sur TOUS les autres PC (même mdp). LAPS génère un mdp unique par device → propagation impossible.
✅ LAPS = solution directe au lateral movement. Question : "Comment empêcher un attaquant de se propager PC à PC via le compte admin local ?" → LAPS.

Patterns récurrents à l'exam

💜 Pattern "Quel est le minimum requis ?"
L'exam pose souvent des questions sur la licence ou la fonctionnalité minimale. Ex: "Quel est le minimum pour bloquer l'accès aux devices non conformes ?" → Entra ID P1 + Intune.
💜 Pattern "BYOD vs Corporate"
Dès que vous voyez "device personnel", pensez MAM. Dès que vous voyez "device d'entreprise avec contrôle total", pensez MDM. Simple mais souvent testée.
💜 Pattern "Dépannage Autopilot"
Questions type : "L'Autopilot échoue, que vérifier ?" → Hardware Hash enregistré ? Profil Autopilot assigné ? Internet disponible ? TPM fonctionnel ? ESP timeout ?
💜 Pattern "Ordre de déploiement"
L'exam teste souvent l'ordre : Root CA → SCEP → Wi-Fi. Ou : Compliance Policy avant Conditional Access. Les ordres sont des pièges classiques.
💜 Pattern "Qui fait quoi : Compliance vs Config Profile vs CA"
Tres souvent testé. Compliance = juge. Config Profile = configure. CA = bloque/autorise. Ces rôles sont distincts et complémentaires.
💜 Pattern "Mode Android Enterprise"
"Tablette de caisse usage unique" → COSU/Dedicated. "PC d'employé perso" → Work Profile. "Smartphone 100% professionnel" → COBO/Fully Managed. "Perso et pro séparés" → COPE.
